title:STM32

01：前言

1.库函数中，cmd函数表示使能，ITConfig表示中断使能，

2.RCC相当于人的心脏，提供频率信号的

3.何为stm32就是一个寄存器占32位，但是在gpio中，四位对应一个引脚，所以gpio共8个引脚，所以需要两个寄存器去对应一个gpioA

4,哪个寄存器寄存到哪个时钟上面，可以参考文档apb2,apb1,ahb三个时钟的各个位，

5.使用地址操作寄存器的时候，&和|的区别：&是进行判断语句，可以看作只对比有1的位置，若两者相同则为1，而|是一种强制置1的语句。所以如果是置1则直接使用|即可，如果想要置0，则需要先使用左移或者右移将对应位置1，然后取反清零同时完成了该数的其他位置1，再将其与源地址&，这样保证了其他位与1&而保持不变，只是需要置0的位置变化了，而不是直接使用0进行&

定时器0/计数器1

定时器个数3个，T0,T1,T2（可多可少，参考官方）

原理：每个一个时间段计数单元就加1，所以单位事件乘以计数就是计时时间

定时器属于CPU内部的资源。

作用：1.计时 2.代替Delay，类似于并行

工作模式

一般只用到模式1 ，晶振为12MHZ，每个电平为1us，

如上图一般只用到模式1 ，模式1工作原理如上图，左侧为计时单元，中间为计数单元，右侧为中断单元，TF0为中断信号。

左侧时钟部分：时钟可以由两部分提供，1.SYSclk系统时钟 2.T0 Pin外设时钟。本51频率为12MHZ，其中12T代表12分频，所以每频为1M，对应的时间为1us。其中C/T代表如果给1，开关连接T0Pin为计数器，如果给0连接计时器SySclk.

中间部分的TL0为低字节，TH0为高字节。共可以存储65535次。

观察定时器相关模式图可以看到图中的开关如何选择，就是通过寄存器来设置的，写代码时就是通过控制寄存器来控制相关功能的。

以下为TCON控制寄存器的详细介绍，它是中断系统里面的东西，参考定时器和中断系统。根据系统图，除了tmod部分，其余都属于tcon部分

TCON可位寻址

TMOD详细图解

GATE=0就是由TR0单独控制，=1就是由INTO和TR0共同控制，一般只用到工作模式1，所以M1,M0=0,1，根据图只有gate金额c/t部分属于tmod，其余都属于TCON部分。

案例：实现使用计时器实现LED等的亮灭。

思路：首先实现计时一秒：因为一个代表1us，65535就是65毫秒，那么怎么使用1ms呢？从64535处开始计时，到65535就是1000个us就是1ms（开始执行中断函数），那么如何计时1s呢，利用if函数判断该函数是否被调用了1000次就是1s

脉冲的原理：默认是共有65535个脉冲，每个脉冲1us，相当于所有脉冲已经编号号并且固定好，当执行的时候仅需要设置从哪里开始执行即可，达到第65535个脉冲是开始执行中断程序，执行完后默认从0开始，而不是从设置的起点开始。

假如说，我们定时50毫秒，TH0，TL0对应着什么值呢？ 上面我们说了，TH0，TL0，组成的16位计数器计数范围是 0—65535 。50ms = 50 000us，我们只要让 TH0 TL0 从（65535 - 50000）开始计数，TH0，TL0就会不停的+1，直到TH0，TL0=65535，就是计数了50000次，时间就是过了50ms。我们只要在中断程序里面，重新赋值给TH0，TL0=(65535 - 50000)，定时器就会不停得每50ms中断一次了。

blazor

Blazor Server 托管模型具有以下优点：（网页文件都是cshtml类型，因为是服务端渲染）

  • 下载项大小明显小于 Blazor WebAssembly 应用，且应用加载速度快得多。
  • 支持瘦客户端。 例如，Blazor Server 应用适用于不支持 WebAssembly 的浏览器以及资源受限的设备。
  • 应用的 .NET/C# 代码库（其中包括应用的组件代码）不适用于客户端。

Blazor Server 托管模型具有以下局限性：

  • 通常延迟较高。 每次用户交互都涉及到网络跃点。
  • 不支持脱机工作。 如果客户端连接失败，应用会停止工作。

Blazor WebAssembly 托管模型具有以下优点：（网页文件都是razor类型，因为是客户端渲染）

  • 没有 .NET 服务器端依赖，应用下载到客户端后即可正常运行。
  • 工作可从服务器转移到客户端。

Blazor WebAssembly 托管模型具有以下局限性：

  • 应用仅可使用浏览器功能。

  • 需要可用的客户端硬件和软件（例如 WebAssembly 支持）。

  • 下载项大小较大，应用加载耗时较长。
